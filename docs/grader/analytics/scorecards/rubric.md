# Repository Grading Rubric

Apply the weights and caps in [methodology](../methodology.md). Evidence must be observable, revision-specific, and linked to stable IDs.

## Product and financial correctness
Fragile: planes or totals can mislead. Foundational: boundaries documented but gaps remain. Operational: all primary values have correct sources. Refined: reconciliation and provenance are tested. Exemplary: explainable, complete, resilient financial truth. Blocker: planning mutates truth or material misleading total.

## Privacy, security, and trust
Fragile: private finance or secrets can leak. Foundational: controls exist with unverified boundaries. Operational: browser/server boundary is tested. Refined: threat review gates features. Exemplary: measurable privacy regression prevention. Blocker: plaintext or secret leakage.

## Accessibility and inclusive interaction
Fragile: core action is unavailable. Foundational: semantic foundations with key gaps. Operational: keyboard and assistive journeys work. Refined: responsive and reduced-preference checks pass. Exemplary: inclusive data exploration is measured. Blocker: inaccessible primary action or core dialog focus failure.

## Core workflow usability
Fragile: users cannot recover key work. Foundational: basic paths work with ambiguity. Operational: imports, review, cashflow, and planning complete reliably. Refined: state, recovery, and progressive detail are consistent. Exemplary: measured low-friction completion.

## Information architecture and comprehension
Fragile: source meaning is unclear. Foundational: groups exist but terminology drifts. Operational: hierarchy and labels explain source. Refined: deep links and cross-links preserve context. Exemplary: users reliably understand financial planes.

## Responsive and Apple-platform adaptation
Fragile: primary content is unusable on target size. Foundational: basic responsive rules. Operational: target viewport behaviors are tested. Refined: iPhone, iPad, and macOS interactions feel deliberate. Exemplary: adaptive presentation preserves efficient density everywhere.

## Visual system and interface consistency
Fragile: inconsistent controls obscure action. Foundational: tokens exist with local drift. Operational: primary states and values are consistent. Refined: provenance, typography, and modes are systematic. Exemplary: precise, accessible, coherent financial presentation.

## Frontend engineering quality
Fragile: repeated logic blocks safe change. Foundational: components are separated. Operational: repeated primitives and state contracts are shared. Refined: performance and fixture boundaries are enforced. Exemplary: new journeys compose safely with minimal drift.

## Backend and data architecture
Fragile: unclear lifecycle risks data. Foundational: active boundaries exist. Operational: active and compatibility surfaces are classified. Refined: migrations are generation-tested. Exemplary: retirement and evolution are routine and reversible.

## Testing and operational reliability
Fragile: important behavior is unverified. Foundational: focused units exist. Operational: finance, privacy, and journey tiers run. Refined: responsive, migration, and deploy checks gate releases. Exemplary: recovery and quality signals are continuously proven.

## Simplicity and cruft control
Fragile: unowned compatibility and dependencies accumulate. Foundational: candidates are inventoried. Operational: deletion has evidence gates. Refined: ownership and drift checks prevent recurrence. Exemplary: lean surfaces retain all supported behavior.

## Innovation and differentiated value
Fragile: speculative features compromise truth. Foundational: opportunities are bounded. Operational: local explainable experiments run. Refined: measured signals improve decisions without privacy loss. Exemplary: privacy-preserving financial intelligence is demonstrably useful.
